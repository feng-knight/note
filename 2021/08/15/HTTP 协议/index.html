用户从输入URL到最终页面展示，这个过程发生了什么？

1) 输入网址敲回车，首先发生页面跳转

浏览器可能已经记住了改地址是否已经重定向，所以需要先判断需不需要 Redirect 以及 Redirect 到哪里

  1. 已经重定向则访问重定向的地址
  2. 没有重定向则访问默认地址
2) 查缓存

请求的资源可能在以前就已经缓存过了，如果存在该资源的缓存，则可以直接从缓存中拿取数据即可

  1. 缓存未过期就直接拿本地缓存
  2. 本地缓存过期了，浏览器会去请求服务器询问本地缓存是否真的过期（Last-Modified时间验证或者Etag数据签名验证）
  3. Last-Modofied或 Etag验证通过则直接拿本地缓存，否则则会发起一次新的数据请求，重新更新本地缓存
3) DNS解析

在没有页面缓存的情况下会对该服务器发起一个新的请求，因为我们在地址栏中输入的是域名，但是域名只有对应到相应的 IP 才能访问到相应的服务器，所以需要做 DNS 解析，将域名解析成对应的 IP 地址

  1. 操作系统会首先检查本地 host 文件下有没有 DNS 映射，有的话就调用这个 IP 地址映射完成域名解析
  2. 没有 host 映射的话，会查找本地 DNS 解析器缓存
  3. 没有本地 DNS 解析器缓存的话就查找本地 DNS 服务器
  4. DNS解析
4) 创建TCP连接，三次握手

只有经过 TCP 的三次握手之后才能真正的创建一个客户端到服务端的TCP连接

三次握手：

为啥不是两次握手而是三次握手？

假设客户端向服务端发起了一个 A 请求，但是 A 请求由于某种原因在半路阻塞了，这时候客户端由于设置了超时时间，在这个时间内客户端没有收到服务端的响应，客户端认为包A丢失了，于是又重新发送了一个请求 B ，服务端成功的响应了请求B，建立连接，两端通信结束后关闭了连接，但是这时候，阻塞的包 A 达到了服务器，服务器不知道这是一个无效的数据包，于是建立了一个无效连接，这时两次握手成功，但是客户端认为自己没有发出任何请求，于是不会对此做出回应，这样就让服务器白白等待，浪费了服务器资源。而在三次握手下，客户端知道自己没有发起这个请求，于是会拒绝这次请求，服务器收到回应也会关闭连接

3. HTTP 缓存：

1) Cache-Control（客户端缓存）
1.1) 缓存时间max-age

设置客户端可缓存，缓存存在的时间为 max-age 设置的值（单位为秒）

即使服务端更新了数据，只要客户端缓存时间没过期，仍然从缓存中拿取数据而不会重新请求服务端

问题来了：假如缓存时间设置的非常长，服务端数据又更新了，难到一直就使用旧数据吗？

解决方案一般是对服务端返回的文件内容进行一个哈希计算，只要文件内容不发生改变，这个哈希值就不会发生变化，通过哈希值生成一个动态码放在请求的静态资源文件名后 ，一旦请求的内容发生改变，对应的 URL 也会相应的改变，从而达到重新请求服务端的效果，从而更新缓存

1.2) 可缓存性
  • public: 任何地方都可以进行缓存
  • private：只有浏览器，发起请求端才可进行缓存
  • no-cache：任何地方都不允许缓存
1.3) 到期
  • max-age=: 缓存的存在时间
  • s-maxage=：只在代理服务器中生效
  • max-stale=：max-age过期后，发起请求方主动带的头，服务端无法设置，在 max-stale 设置的时间内，就算max-age 过期，发起请求方仍可以从缓存中读取
1.4) 重新验证
  • must-revalidate：当max-age过期后，客户端必须重新去服务端请求数据来验证是否真的过期，而不能直接从缓存中拿
  • proxy-revalidate：用于缓存服务器
1.5) 其他
  • no-store：本地与代理服务器都不可进行缓存，每次拿资源都必须从服务器请求

浏览器查找缓存原理：

浏览器在创建一个请求时会首先在本地缓存中找，有本地缓存则直接从本地缓存中取，没有的话则会去代理服务器中找是否有代理缓存，有则从代理缓存中拿，两者都没有才会重新去源服务器去请求

2) 请求验证
  • Last-Modified：上次修改时间，配合If-Modified-Since和If-Unmodified-Since使用

浏览器在请求资源时服务端在返回资源时会带上Last-Modified这个头，然后客户端每次在请求服务端的时候都会带上If-Modified-Since这个头与Last-Modified这个时间作比较，若一样则说明资源没有更改，服务器会反馈给客户端可以从缓存中取，否则则说明资源被修改了，需要重新请求

  • Etag：数据签名，配合 If-Match和 If-No-Match

对资源的内容会产生一个唯一的数据签名（对内容进行哈希计算），当资源被修改后这个数据签名也会相应的发生变化

我们为资源标记这个数据签名，下次浏览器在请求资源的时候就会带上If-Match或If-Non-Match这个头，这个头的值就是服务端返回的Etag的值，我们对比服务器中现在存在的数据签名与浏览器传过来的签名，看是否一致，一致则说明资源未修改，可以使用缓存，反之则说明资源被修改了，需要重新请求

3) 一些状态码

  • 304 Not Modified

    语义：资源未修改，可以直接读缓存

  • 302 Found

    语义：临时重定向

  • 301 Moved Permanently

    语义：永久重定向

4. Web 安全及防范(xss/csrf)：

XSRF/CSRF 定义

XSRF/CSRF 跨站请求伪造，是一种对网站的恶意利用，听起来有点像跨站脚本攻击XSS,但是与XSS非常不同，XSS利用站点内的信任用户，而CSRF则通过伪装成信任用户来请求受信任的网站，与XSS相比，CSRF更具危险性。

攻击方式：

XSRF攻击，攻击者利用可以在用户的网站中注入脚本或通过 URL 的形式引导用户向攻击者的服务器发送用户自己的 Cookie ，从而攻击者可以利用用户发送过来的 Cookie 来得到用户在该网站的个人数据。

XSS(Cross Site Scripting)攻击

叫 XSS 不叫 CSS 是为了不想和层叠样式表(CSS) 弄混淆，故意缩写为 XSS,攻击者向 WEB 页面中插入恶意的 HTML 代码或者 JavaScript 代码以此来盗用用户信息

防范方法：

  1. 避免在 cookie 中泄漏用户信息
  2. 设置 HttpOnly 来防止 javascript 获取 cookie
  3. 表单提交尽量采用 POST 而非 GET
  4. 在源头控制，把输入的一些不合法的东西都过滤掉，从而保证安全性。如移除用户提交的的DOM属性如onerror，移除用户上传的Style节点，,，``节点等
  5. 对标签进行转码
