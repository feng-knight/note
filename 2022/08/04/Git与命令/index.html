Git与命令

发表于 2022-08-04

1. 创建SSH密钥，附上：SSH教程

2. 将仓库拉到本地： (如果是私有仓库，需要别人邀请你呦！)

拉取origin远程仓库：

git clone (仓库URL)

然后你会发现你的Desktop上多了一个文件夹 (没错，就是那个仓库啦！)

注：最好把你项目创建在这个文件夹中，这样你在写项目的过程中更新了哪些代码看的很清楚。

3. 提交代码：

注：你不能直接在master分支上提交代码 (如果是别人创建的仓库，你可能连提交的权限都没有)，需要自己在创建一个分支。

1) 在仓库文件夹中执行gitbash：

2) 创建新的分支：git branch 分支名

3) 切换到你创建的分支上：git checkout 分支名

4) 查看状态：git status

5) 将你本地的修改提交到暂存区：

git add .(后面有个“点”，你别跳过了！！！)

git commit -m '' (后面引号的内容是有规范的)

附上：git commit 提交规范

① git commit -m 'docs: update README.md'

② git commit -m 'test'

6) 推送分支：git push

在推送的时候你可能会出现一些问题：

没关系，直接在执行一次它给你的这个命令就OK了！

然后直接刷新GitHub在你创建的分支上就能看见你更新的代码了。

注：最好在执行完每一步后，都执行以下git status，看看它推荐你下一步做什么，防止你自己操作失误！

4. 有关项目文件夹和仓库文件夹位置及其他问题的注意点：

  • 如果你的项目文件夹和仓库处于同一等级（也就是同时处在桌面），那你可以将项目文件夹里的所有文件复制到仓库文件夹中

    • 肯定会出现文件重复问题：.git文件夹保留仓库里的，README.md保留项目里的。

  • 现在仓库里的.gitignorn文件可以设置推送时过滤的文件（因为node_modules文件夹太大，也不需要推送到远程GitHub）

    • .DS_Store
node_modules
/dist

  • 再vscode中的源代码管理可以看见更改的文件：

    • M 代表修改了这个文件
    • U 代表新增的文件

5. 从远程仓库拉代码：

  • 说明：这个步骤是为了防止一个人以上开发会产生代码冲突，你不做这一步的话看不到别人为这个项目做的工作，也就是不知道别人在之前的项目代码上已经更新了哪些页面和功能，你也无法继续在别人的更新上继续做开发。

  • 步骤：

    • 将远程master代码拉倒本地仓库：git pull origin/master

    • 将远程master代码合并到你的分支：git merge origin/master

      对于合并代码，GitHub也有相应的操作，也可以查看你的分支与master主分支的是否有不相同的地方，这个自己百度去吧。或者我以后有时间再去介绍

    • 若前两个步骤没有产生冲突（conflict）的话，你可以直接进git pull，相当于直接进行了上面两步

6. 回滚代码：

  • 如果你在提交代码或者合并代码时出现了错误（可能是之前的代码被你覆盖，而你提交的代码又不是最新的等等情况），这时就需要你学会回滚自己的代码了。
  • 查看历史记录：git log（复制你想退回的你需要的历史代码那里很长一段的历史记录码）
  • 然后执行 git reset --hard 你刚才复制的历史记录码
  • 最后 git pull就ok了。
  • 注：注意多执行git status，看看它推荐你下一步做什么，防止你自己操作失误！

8. git 清除、记住用户名和密码

清空所有用户名和密码：

git config --system --unset credential.helper

清除缓存的用户名和密码：

git credential-manager uninstall

记住密码：

git config --global credential.helper store

9. git cherry-pick 教程

对于多分支的代码库，将代码从一个分支转移到另一个分支是常见需求。

这时分两种情况。一种情况是，你需要另一个分支的所有代码变动，那么就采用合并（git merge）。另一种情况是，你只需要部分代码变动（某几个提交），这时可以采用 Cherry pick。

三、配置项

git cherry-pick命令的常用配置项如下。

（1）-e，--edit

打开外部编辑器，编辑提交信息。

（2）-n，--no-commit

只更新工作区和暂存区，不产生新的提交。

（3）-x

在提交信息的末尾追加一行(cherry picked from commit ...)，方便以后查到这个提交是如何产生的。

（4）-s，--signoff

在提交信息的末尾追加一行操作者的签名，表示是谁进行了这个操作。

（5）-m parent-number，--mainline parent-number

如果原始提交是一个合并节点，来自于两个分支的合并，那么 Cherry pick 默认将失败，因为它不知道应该采用哪个分支的代码变动。

-m配置项告诉 Git，应该采用哪个分支的变动。它的参数parent-number是一个从1开始的整数，代表原始提交的父分支编号。

上面命令表示，Cherry pick 采用提交commitHash来自编号1的父分支的变动。

一般来说，1号父分支是接受变动的分支（the branch being merged into），2号父分支是作为变动来源的分支（the branch being merged from）。

四、代码冲突

如果操作过程中发生代码冲突，Cherry pick 会停下来，让用户决定如何继续操作。

10. Git 解决 fatal: not in a git directory

在添加 git 邮箱和账号时会出现 fatal: not in a git directory
下面就是来交大家如何快速解决这个问题。

1) 先打开Git Bash所在的文件夹

2) 双击打开git-cmd.exe

3) 先输 git config –global user.email 你的邮箱号
4) 再输 git config –global user.name 账号名
5) 分别回车后即可解决not in a git directory问题
6) 切记一定要先在cmd中先输入邮箱再输入账号，否则依旧会报错
结语：实质上第一次报错是因为目录结构不对，Git Bash图中顶上的 MINGW64 ~ 是系统根目录 ，我们需要切换至Git 目录，就不会报错了。

13. Hexo部署Github的用户名和密码问题

如今的hexo博客hexo d时会提示输入github的用户名和密码，但是输入之后提示无效，网上查了一些文章说是github已经无法再用密码登录推送，只能用token，所以要重新设置一下。

① 生成Token：

首先打开github settings中的Developer settings，选择Personal access tokens中的Tokens(classic)，然后Generate new token(classic)。若只需要读取仓库的权限，可以只勾选 “repo” 相关的权限。生成Token之后，切记！！！要复制记录下来，因为后面你就看不到了！
